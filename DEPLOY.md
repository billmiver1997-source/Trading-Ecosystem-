# Rebuilding this from scratch

This is the full recipe to stand the whole Trading Nova ecosystem back up on a
fresh Ubuntu server (Hostinger VPS or otherwise), in case the current server
is ever lost, wiped, or you want to move providers. The code itself lives on
GitHub — this file is everything *around* the code that GitHub doesn't track.

## 1. Get a server

Any Ubuntu 22.04/24.04 VPS with at least 2GB RAM works — this whole ecosystem
is lightweight (19 Python processes, mostly idle between scheduled runs).

## 2. Clone the repo

```bash
cd /root
git clone https://github.com/billmiver1997-source/Trading-Ecosystem-.git tradingbot
cd tradingbot
```

## 3. Create the virtual environment

The systemd unit files below assume the venv lives at `/root/tradingbot`
itself (Python 3.12):

```bash
apt update && apt install -y python3.12-venv
python3 -m venv /root/tradingbot
```

## 4. Install dependencies

```bash
/root/tradingbot/bin/pip install -r requirements.txt
```

## 5. Set up environment variables

```bash
cp .env.example .env
nano .env   # fill in every value — see .env.example for what each one is
```

You'll need, at minimum:
- 3 Telegram bot tokens from @BotFather (Signal bot, Main VA bot, News bot)
- The 3 channel IDs those bots post to
- Your own Telegram user ID (`OWNER_ID` / `ADMIN_CHAT_ID`)
- An Anthropic API key with credit — **every AI-driven feature (Analysis,
  news summaries, sentiment, education, tips) silently degrades to a
  fallback message without this**

## 6. Download the stock images

```bash
/root/tradingbot/bin/python3 setup_images.py
```

Downloads the rotating stock-photo pools (news/tips/psychology/session
images) from Unsplash into `/root/tradingbot/images/`. Safe to re-run —
skips files that already exist.

## 7. Install the systemd services

All 19 unit files are in `deploy/systemd/`. Copy them into place and enable:

```bash
cp deploy/systemd/*.service /etc/systemd/system/
systemctl daemon-reload
for f in deploy/systemd/*.service; do
  svc=$(basename "$f")
  systemctl enable --now "$svc"
done
```

Check everything came up:

```bash
systemctl list-units --type=service --all | grep -E 'nova|tradingbot'
```

## 8. Set up the two cron jobs

```bash
crontab -e
```

Add the two lines from `deploy/crontab.txt`:
- `*/5 * * * *` runs `monitor.sh` — the watchdog that restarts any crashed
  service and DMs you (`ADMIN_CHAT_ID`) when it does
- `*/10 * * * *` pulls the latest code from `main` — **this updates the
  files on disk but does NOT restart running services**; a code change only
  takes effect after `systemctl restart <service>`. `monitor.sh` does not
  do this for you.

## 9. Verify

```bash
systemctl list-units --type=service --state=failed --no-legend
```

Should be empty. Then check a couple of logs for startup errors:

```bash
journalctl -u nova_listener.service -n 20 --no-pager
journalctl -u tradingbot.service -n 20 --no-pager
```

## What's NOT in this recipe (by design)

- **Runtime data** (`users.json`, `journal.json`, `trade_stats.json`,
  `open_trades.json`, personal journals, cursor/state files) — these are
  live application state, not deployment config. They start empty on a
  fresh server. If you need to preserve trade history / subscriber lists
  across a migration, copy those specific `.json` files from the old server
  separately.
- **The `.env` file itself** — never committed, contains live secrets.
  `.env.example` is the template; you fill in real values by hand each time.
- **`bin/`, `lib/`, `lib64/`, `share/`, `pyvenv.cfg`** — these are the venv
  itself, regenerated by steps 3-4, never committed.
